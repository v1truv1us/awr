# AWR Browse Quality Plan

## Goal

Improve `awr browse <url>` so it defaults to a **content-first** browser presentation instead of a raw DOM/text dump.

The next pass should focus on these six improvements:
1. render from the active DOM snapshot instead of reparsed raw HTML
2. remove the giant references block in browse mode
3. show links as selectable inline items plus footer preview only
4. add page landmarks and spacing for `header` / `nav` / `main` / `aside` / `footer`
5. suppress or collapse boilerplate-heavy regions by default
6. add a reader-style main-content heuristic where helpful

## Current State

What is causing the bad output now:
- `src/browser.zig` renders browse pages from `result.html`, not from page state after load
- `src/render.zig` is still optimized for static text output with inline `[n]` references and a `References:` section
- `ScreenModel` only gives line-oriented link metadata, not enough for inline presentation
- landmarks are flattened into generic block rendering
- there is no boilerplate suppression or main-content selection today

Important constraint:
- `src/dom/bridge.zig` explicitly says JS DOM mutations are **not** reflected back into the Zig DOM tree
- so a true implementation of step 1 is not ready today without bridge/snapshot work

## Recommended Approach

Treat this pass as a **content-first browse renderer pass** with one staged seam for future active-DOM rendering.

Meaning:
- improve browse quality now using the parsed Zig DOM plus heuristics
- add a render-source abstraction in `src/page.zig`
- defer the final switch to a true active DOM snapshot until the bridge can produce one reliably

## Implementation Order

### 1. Add a browse render profile and render-source seam

Files:
- `src/page.zig`
- `src/render.zig`
- `src/browser.zig`

Changes:
- add a browse-oriented render entrypoint in `src/page.zig`
- stop calling `renderHtmlModel(... result.html ...)` directly from `src/browser.zig`
- introduce explicit browse render settings in `src/render.zig`, such as:
  - `profile: .content_first | .full_page`
  - `link_style: .inline | .reference`
  - `collapse_boilerplate: bool`
  - `show_references: bool`
- default browse mode to `profile = .content_first`

Why first:
- this creates one place to route later active-DOM rendering without rewriting browser mode twice

### 2. Remove browse-mode references block and footnote-style links

Files:
- `src/render.zig`
- `src/browser.zig`

Changes:
- make `References:` emission optional
- disable it in browse mode
- stop appending `[n]` footnote markers in browse mode
- keep existing reference-style rendering intact for one-shot non-browse output if useful

Desired result:
- browse mode shows readable inline link text
- footer keeps the selected link URL preview
- giant reference appendix disappears from normal browsing

### 3. Enrich screen metadata just enough for better browse presentation

Files:
- `src/render.zig`
- `src/browser.zig`

Changes:
- extend `ScreenLink` enough to support future inline emphasis if easy
  - at minimum keep stable label/text metadata
  - ideally add span coordinates later, but do not force a TUI rewrite in this pass
- keep `src/browser.zig` line-oriented
- continue using footer preview as the main link target display

Scope note:
- inline highlighting is optional in this pass
- inline link rendering itself is required

### 4. Add landmark-aware rendering and spacing

Files:
- `src/render.zig`

Changes:
- render `main`, `article`, `section` with stronger spacing and priority
- render `header`, `nav`, `aside`, `footer` with lighter treatment
- add subtle labels or separators where they improve comprehension
- make browse-mode output feel like grouped page regions instead of one flat text stream

Recommended default behavior:
- `main` / `article`: full render
- `header` / `nav` / `aside` / `footer`: compact treatment, often collapsed in content-first mode

### 5. Add boilerplate suppression and collapse rules

Files:
- `src/render.zig`
- `src/page.zig`
- add `src/browse_heuristics.zig`
- optionally small helpers in `src/dom/node.zig`

Changes:
- add a small heuristic module for browse content policy
- start with obvious cheap signals only:
  - tags: `nav`, `footer`, `header`, `aside`
  - roles: `main`, navigation-like roles
  - class/id tokens: `nav`, `menu`, `footer`, `header`, `sidebar`, `related`, `breadcrumb`, `social`, `promo`, `newsletter`
  - link density
  - text density
- in `content_first`, collapse or skip boilerplate-heavy subtrees by default
- render compact placeholders only where useful, e.g. `[nav collapsed]`

Rule for this pass:
- favor false negatives over false positives
- better to leave some noise than hide the main content

### 6. Add reader-style main-content selection

Files:
- `src/browse_heuristics.zig`
- `src/render.zig`
- `src/page.zig`
- possibly `src/dom/node.zig`

Changes:
- choose a preferred content root before rendering in browse mode
- use this priority order:
  1. `<main>`
  2. `<article>`
  3. `[role=main]`
  4. highest-scoring container by simple heuristic
- scoring inputs:
  - positive: text length, paragraph count, punctuation/sentence density
  - negative: link density, boilerplate tokens, explicit nav/footer/header/aside landmarks
- if no candidate scores well, fall back to full body with the same improved landmark/collapse behavior

## Step 1 Reality Check: Active DOM Snapshot

Step 1 is the only item not fully ready today.

Current blocker:
- `src/dom/bridge.zig` does not maintain a Zig DOM that reflects JS mutations
- current JS bridge objects are good enough for script compatibility, but not yet a trustworthy post-script render source

Plan for this pass:
- add a render-source seam in `src/page.zig`
- keep browse rendering on parsed DOM for now
- do **not** claim true active-DOM rendering yet

Follow-up once this pass lands:
- improve the bridge or add a reliable JS-side serializer
- then switch browse rendering from parsed DOM to active snapshot

## Verification Plan

### Render tests

In `src/render.zig`:
- browse profile does not emit `References:`
- browse profile renders links inline while still populating `ScreenModel.links`
- landmark spacing improves ordering/readability
- boilerplate-heavy nav/footer regions are collapsed or omitted in `content_first`

### Heuristic tests

In `src/browse_heuristics.zig`:
- `<main>` beats sibling nav/footer noise
- `<article>` beats link-heavy sidebars
- `[role=main]` beats generic wrappers
- high-link-density blocks are penalized
- simple article-only pages are not over-filtered

### Page tests

In `src/page.zig`:
- browse render path uses the new render seam instead of reparsing raw HTML directly in `src/browser.zig`
- relative links still resolve correctly in rendered browse output
- body/content extraction and browse rendering stay aligned on the same content-root selection logic

### Manual verification

Use local deterministic HTML fixtures first, then re-test hostile real pages like Google.

Manual checks:
1. `awr browse <fixture>` opens with no `References:` block
2. link text is visible inline and footer previews selected target
3. obvious nav/footer/legal boilerplate is suppressed or collapsed
4. article/main content is shown first in content-first mode
5. fallback still shows a readable full page when no good main-content candidate exists
6. search and link navigation still work after the render changes

## What To Defer

Do not expand scope in this pass to:
- full DOM mutation reflection into Zig DOM
- a TUI rewrite
- mouse support
- form UX
- click simulation / richer browser events
- visual CSS/layout rendering

## Bottom Line

The right next pass is:
- make browse mode content-first by default
- remove debug-style link references from normal browsing
- add landmarks and boilerplate suppression
- add a simple reader heuristic
- stage, but do not fully promise yet, the future switch to a true active-DOM snapshot

That is the shortest path from the current browser prototype to output that actually feels useful.
