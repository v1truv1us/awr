# Remaining-Work Spec Consolidation Plan

## Goal

Make `spec/MVP.md` the canonical umbrella spec for all remaining work after the shipped WebMCP v1 slice, and back it with a small set of sub-specs that clearly separate:

- active work that should be executed now;
- deferred work that should stay documented but not be worked on now;
- historical docs that are no longer authoritative.

## What This Plan Is Solving

- `MVP_PLAN.md` is a shipped record, not an active plan.
- `spec/MVP.md`, `spec/PRD.md`, `README.md`, and `MVP_BACKLOG.md` currently make conflicting status claims.
- `src/main.zig` clearly ships `awr tools`, `awr call`, and `awr mock`, while `src/mcp_stdio.zig` is present but incomplete/stale and not part of the active CLI surface.
- Browser/TUI and later fingerprinting work are still important, but they should be documented as deferred rather than mixed into current execution scope.

## Recommended Doc Structure

### Canonical umbrella

- Rewrite `spec/MVP.md` as the single source of truth.

Recommended sections:

1. Purpose and authority
2. Current shipped baseline
3. Current execution scope
4. Active remaining work
5. Sub-spec index
6. Deferred roadmap (explicitly not for current execution)
7. Verification matrix
8. Document governance / status rules

### Active sub-specs

- Add `spec/subspecs/mvp-remainder.md`

This should absorb the actionable remainder from `MVP_BACKLOG.md` and define the current workstreams that are supposed to be executed now:

- primary demo quality;
- coverage depth;
- verification gaps;
- doc consolidation.

For each workstream, include:

- exact file targets;
- acceptance criteria;
- verification commands;
- explicit non-goals.

### Deferred sub-specs

- Add `spec/subspecs/mcp-stdio.md`
- Add `spec/subspecs/browser-tui.md`
- Keep `spec/Fingerprint-Plan.md`, but rewrite its header and framing so it is clearly future-only and not current execution

These docs should stay linked from `spec/MVP.md`, but each must open with a banner stating that it is roadmap material and not part of the active work queue.

## Recommended Changes By File

### `spec/MVP.md`

Rewrite it into the umbrella document instead of treating it as a stale stub-closure checklist.

Key rules for the rewrite:

- preserve only status claims that are re-verified against current code/docs;
- move active execution details out to `spec/subspecs/mvp-remainder.md`;
- keep future work visible, but only in a clearly labeled deferred section;
- state that `MVP_PLAN.md` is historical and that `spec/PRD.md` is product context, not execution authority.

### `spec/subspecs/mvp-remainder.md`

Use this as the execution spec for current work.

Initial recommended workstreams:

1. Demo quality / primary path readability
2. WPT + Test262 curated coverage growth
3. Verification closures (for example byte-level TLS proof / pool stress coverage if still applicable)
4. Doc consolidation and stale-reference cleanup

### `spec/subspecs/mcp-stdio.md`

Document the intended target state for `src/mcp_stdio.zig`, `src/main.zig`, and `build.zig`, but mark it deferred.

Minimum sections:

- scope;
- current status;
- target CLI / protocol surface;
- critical files;
- acceptance tests;
- explicit note: not active now.

### `spec/subspecs/browser-tui.md`

Document the target state for `src/browser.zig`, `src/tui.zig`, `src/render.zig`, and `src/browse_heuristics.zig`, but mark it deferred.

Minimum sections:

- scope;
- current status;
- browse command wiring expectations;
- rendering / navigation acceptance criteria;
- explicit note: not active now.

### `spec/Fingerprint-Plan.md`

Keep this file to minimize churn, but remove the stale "Phase 3 / not yet started" framing and convert it into a future-roadmap spec.

It should explicitly say:

- this work is not part of the active remainder;
- it should not be picked up until the active remainder in `spec/subspecs/mvp-remainder.md` is complete;
- it exists to preserve design intent, not to define current execution priority.

### `MVP_PLAN.md`

Add a clear historical banner at the top:

- shipped v1 WebMCP slice;
- not the active remaining-work spec;
- superseded by `spec/MVP.md` for current execution.

### `spec/PRD.md`

Keep it as product framing only.

Update its banner/status so it no longer reads as the active implementation plan. Point readers to `spec/MVP.md` for execution scope and to the sub-specs for track-level details.

### `MVP_BACKLOG.md`

Fold active content into `spec/subspecs/mvp-remainder.md`, then either:

- mark it superseded and keep it for history; or
- archive/delete it after references are updated.

### `README.md` and `AGENTS.md`

Update spec references so the repo consistently points to:

- `spec/MVP.md` as canonical;
- `spec/subspecs/mvp-remainder.md` for active execution;
- deferred sub-specs as roadmap only.

Also fix stale references to missing phase-plan paths.

## Execution Order

1. Audit current status claims before copying them forward.
2. Rewrite `spec/MVP.md` into the umbrella spec.
3. Create `spec/subspecs/mvp-remainder.md` from the actionable remainder.
4. Create deferred sub-specs for MCP stdio and browser/TUI.
5. Rewrite `spec/Fingerprint-Plan.md` as future-only roadmap material.
6. Add historical/superseded banners to `MVP_PLAN.md`, `spec/PRD.md`, and `MVP_BACKLOG.md`.
7. Update `README.md` and `AGENTS.md` references.

## Verification

### Doc-level verification

- `spec/MVP.md` is the only file that claims to be canonical.
- Every active remaining-work item appears in exactly one active sub-spec.
- Deferred work appears only under clearly labeled roadmap sections/docs.
- `MVP_PLAN.md`, `spec/PRD.md`, and `MVP_BACKLOG.md` all explicitly state their non-canonical status.
- No doc still claims that `spec/MVP.md` and `MVP_BACKLOG.md` are both the active source of truth.

## Recommended Outcome

After this doc pass, the repo should have:

- one canonical umbrella spec: `spec/MVP.md`;
