# AWR Phase 1 Closure Plan

## Goal

Close Phase 1 on the current owned BoringSSL + owned client path. Do not reopen work that is already in the codebase.

## Current Read

- `src/net/tls_awr_shim.c` already offers `h2` then `http/1.1` and already enables ALPS.
- `src/client.zig` already routes HTTPS through `fetchHttpsOwned()` and dispatches by negotiated ALPN.
- `src/net/h2_shim.c` already sets Chrome-like SETTINGS, WINDOW_UPDATE, and pseudo-header order.
- The real remaining gaps are verification depth, redirect e2e coverage, and connection-pool integration.
- Repo docs disagree with each other. Treat that as stale merged-doc drift, not as separate sources of truth.

## Remove From The Old Plan

- Do not spend time re-enabling H2 ALPN in `src/net/tls_awr_shim.c`.
- Do not spend time re-enabling ALPS in `src/net/tls_awr_shim.c`.
- Do not spend time re-wiring ALPN dispatch in `src/client.zig`.
- Do not spend time fixing H2 pseudo-header order unless a new verification test proves a live-path mismatch.

## Recommended Execution Order

1. Tighten TLS closure verification in `src/net/tls_conn.zig`.
   - Replace the current loose `tls.peet.ws` substring checks with structured assertions against the returned JSON.
   - Record the exact JA4 target produced by the current owned stack and assert it explicitly.
   - Keep the HTTP/1.1-only ALPN override for the `tls.peet.ws` test if that is still the simplest way to read the response body.

2. Add closure-grade HTTPS/H2 e2e coverage in `src/test_e2e.zig` and small test hooks in `src/client.zig` if needed.
   - Keep the existing owned-stack HTTPS success proof for `https://example.com/`.
   - Add a real H2 fetch test against a stable H2 endpoint and assert: status `200`, non-empty body, expected body marker, and negotiated ALPN `h2`.
   - If the client does not currently expose the negotiated protocol to tests, add the smallest possible test-only surface to observe it.

3. Add first-flight H2 frame verification in `src/net/h2session.zig` and/or `src/net/h2_shim.c`.
   - Capture outbound bytes through the existing send callback path.
   - Assert the first SETTINGS tuple matches the Chrome-like values already encoded in the repo.
   - Assert the connection-level WINDOW_UPDATE increment is `15663105`.
   - Assert the submitted pseudo-header order remains `:method, :authority, :scheme, :path`.

4. Integrate real pool reuse in `src/client.zig` and `src/net/pool.zig`.
   - Wire `acquireIdle`, `addNew`, and `release` into the live client flow instead of only checking limits.
   - Fix total-count and close/eviction accounting where needed so the pool cannot drift.
   - Prefer the smallest correct scope first: HTTP and HTTPS HTTP/1.1 fallback reuse before any larger H2 session-reuse design.

5. Add redirect-chain e2e coverage in `src/test_e2e.zig`.
   - Success case: follow a 3-hop redirect chain.
   - Failure case: exceed `max_redirects` and assert `TooManyRedirects`.
   - Exercise the owned HTTPS path, not only option-level unit tests.

6. Reconcile the merged docs after code and tests are green.
   - Update `PHASE1_EXIT_STATUS.md` to reflect the current codebase, not the older pre-owned-stack story.
   - Update `PHASE1_CLOSURE_PLAN.md`, `TLS_RESUME_PLAN.md`, `CLAUDE.md`, `spec/Phase1-Networking-TLS.md`, and `spec/PRD.md` so they all describe the same Phase 1 contract.
   - Remove any remaining `awr` vs `awr-spec` split-brain language.
