## Summary

The plan correctly identifies the core problem (link-density heuristic not applied in the renderer) and proposes the right architectural fix (detect link-list tables in `renderTable` and skip separators). However, there is a **critical density metric bug** in Change 2 and the existing test fixture does not actually exercise the separator code path.

## 1. Does the plan correctly identify the problem? Is width clamping actually working already?

**Verdict: Correct.**

Change 1 analysis is accurate. Tracing through `renderTable` (src/render.zig):

- Lines 797–814: Widths are measured using `measureWrappedWidth(text, per_col_width)`, which wraps at `per_col_width`.
- Lines 815–817: `@max(@min(width.*, per_col_width), 1)` clamps each width.
- Lines 821–828: Separator dashes use `widths[ci] + 2`, which operates on already-clamped values.

Separators can still exceed `max_width` only if `max_cols * per_col_width + max_cols > max_width`, but `per_col_width` is computed from `tableColumnWidthLimit` (line 794, definition at 904–911) exactly to prevent this. The clamping is correctly placed.

**No change needed for Change 1. The plan is right to say so.**

The real root cause is entirely Change 2.

## 3. Will the link-list detection logic work correctly for HN's table structure?

**Verdict: NO — critical density metric bug.**

The proposed `isLinkListTable` uses `countTableLinkMetrics`, which computes:

```zig
const density = @as(f64, @floatFromInt(link_count)) / @as(f64, @floatFromInt(text_bytes));
return density >= 0.35;
```

This is **link count / total text bytes**, not link text bytes / total text bytes.

**Working through the test fixture:**

The HN fixture has:
- 4 title rows: links like "Alpha story with a long title..." (~80 chars each)
- 4 metadata rows: "42 points by alice 2 hours ago | [12 comments]" (~51 chars, link is ~13 chars)

Total `text_bytes` (non-whitespace): ~4×80 + ~4×51 = ~524 chars
Total `link_count`: 6

Density = 6/524 ≈ **0.011** — far below the 0.35 threshold.

The `link_density` condition in `browse_heuristics.zig:155` uses a **different metric**: `link_text_bytes / text_bytes`. For HN tables, `link_text_bytes` ≈ title text (~320 chars), giving density ≈ 0.61.

**Fix required**: `countTableLinkMetrics` must mirror `collectMetrics` from `browse_heuristics.zig:171–193`, tracking `link_text_bytes` (characters inside `<a>` tags) separately from `text_bytes` (all non-whitespace characters), then computing `link_text_bytes / text_bytes`.

**Suggested corrected threshold**: Use `link_text_bytes / text_bytes >= 0.35` to match `metrics.linkDensity()` from browse_heuristics. The minimum link/text byte counts (6 links, 80 text bytes) should also be kept from `scoreLinkListContainer`.

## 4. Are the test cases sufficient?

**Verdict: NO — existing HN fixture doesn't exercise the separator path.**

The test at `render.zig:1347` ("HN-like table layout respects max width") has this structure:

```html
<table><tr><td>
  <table>
    <tr><td><a>Alpha story...</a></tr>
    <tr><td>42 points by alice... | <a>12 comments</a></tr>
    <tr><td><a>Beta story...</a></tr>
    <tr><td>21 points by bob... | <a>8 comments</a></tr>

```

The **outer `<table>`** has exactly **one `<tr>`** (lines 1349–1356 only have one `<tr>` directly under the outer `<td>`). Therefore, `collectTableRows` on the outer table returns 1 row, and in `renderTable` the separator block at line 822 (`if (ri > 0)`) is never entered — no separators are generated regardless of whether the fix is applied.

The existing test asserts `indexOf(model.text, "----------------------------------------") == null` and it passes — but it passes because the fixture structure means no separator is ever generated, not because the fix works. This test does not validate the bug fix.

The proposed new tests (link-list table omits separators, regular table keeps separators) are correct in intent but inherit the same structural flaw if the fixture isn't corrected.

## 5. Risks and Edge Cases

### Edge cases the plan misses:

**a) Tables with mixed content density**
A table where some rows have high link density and others have low (e.g., HN-style: title rows vs metadata rows). The current heuristic aggregates over the entire table, which may be correct — but if a table has many metadata rows with low link density, the aggregate density could drop below threshold even though title rows are high-density. The test fixture should cover this (it does have both title and metadata rows).

**b) Single-row link-list tables**
A table with only 1 row that would qualify as a link-list table. The separator code path won't run (`ri > 0` never true), but the content will still render. This is fine — no separator = no overflow.

**c) Browse mode coupling**
`chooseContentRoot` → `bestLinkListContainer` already identifies HN's inner stories `<table>` as the content root (confirmed by the test at `browse_heuristics.zig:327–344`). When `renderBrowseModel` is called, `renderTable` will be invoked on the stories `<table>`, so the detection will work. However, if `render` (non-browse) is called directly on an HN page, the outer wrapper table won't be identified as a link-list container by browse_heuristics, and the renderer heuristic will need to work on the wrapper table too. The proposed approach is correct — the render-level heuristic should be self-contained.

**d) Zero `<tr>` or zero `<td>` tables**
`renderTable` already handles these via early returns at lines 781 and 792. No issue.

**e) `is_link_list` computed before early returns**
The plan places `const is_link_list = isLinkListTable(elem)` before the row loop. This is correct — `elem` is the `<table>` element passed to `renderTable` and is valid regardless of early returns on line 781 (`rows.items.len == 0`) or 792 (`max_cols == 0`).

## Required Corrections

### CRITICAL — Fix density metric in Change 2

Replace `countTableLinkMetrics` with a version that tracks `link_text_bytes` separately:

```zig
fn isLinkListTable(elem: *const dom.Element) bool {
    var link_count: usize = 0;
    var text_bytes: usize = 0;
    var link_text_bytes: usize = 0;
    var paragraph_count: usize = 0;
    countTableLinkMetrics(elem, &link_count, &text_bytes, &link_text_bytes, &paragraph_count);
    if (link_count < 6) return false;
    if (text_bytes < 80) return false;
    if (paragraph_count > 0) return false;
    const density = if (text_bytes == 0) 0.0 else @as(f64, @floatFromInt(link_text_bytes)) / @as(f64, @floatFromInt(text_bytes));
    return density >= 0.35;
}

fn countTableLinkMetrics(
    elem: *const dom.Element,
    link_count: *usize,
    text_bytes: *usize,
    link_text_bytes: *usize,
    paragraph_count: *usize,
) void {
    countTableLinkMetricsInner(elem, false, link_count, text_bytes, link_text_bytes, paragraph_count);
}

fn countTableLinkMetricsInner(
    elem: *const dom.Element,
    in_link: bool,
    link_count: *usize,
    text_bytes: *usize,
    link_text_bytes: *usize,
    paragraph_count: *usize,
) void {
    const is_link = eql(elem.tag, "a");
    if (is_link) link_count.* += 1;
    if (eql(elem.tag, "p")) paragraph_count.* += 1;

    const child_in_link = in_link or is_link;
    for (elem.children.items) |child| {
        switch (child) {
            .text => |text_node| {
                for (text_node.data) |ch| {
                    if (!std.ascii.isWhitespace(ch)) {
                        text_bytes.* += 1;
                        if (child_in_link) link_text_bytes.* += 1;
                    }
                }
            },
            .element => |child_elem| {
                countTableLinkMetricsInner(child_elem, child_in_link, link_count, text_bytes, link_text_bytes, paragraph_count);
            },
            else => {},
        }
    }
}
```
